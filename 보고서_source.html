1. 프로젝트 개요

1.1 배경 및 목적

국립중앙도서관은 '사서문답' 서비스를 통해 이용자 질문에 전문 사서가 직접 답변하는 Q&A 데이터를 5,900건 이상 축적하고 있습니다. 그러나 이 데이터가 검색 가능한 형태로 체계화되지 않아 이용자가 원하는 정보를 효율적으로 찾기 어려운 상황이었습니다.

본 프로젝트는 이 Q&A 데이터를 RAG(Retrieval-Augmented Generation) 방식으로 구조화하여, 자연어 질문 입력 시 유사 Q&A를 자동 검색하고 Gemini AI가 이를 바탕으로 답변을 생성하는 AI 사서 응답시스템을 구축하는 것을 목표로 합니다.

2. 개발 단계별 진행 과정

Phase 1 — 기획 및 설계

초기에는 OpenAI GPT-4 + text-embedding-ada-002 조합을 계획하였으나 API 크레딧 부족으로 대안을 마련하였습니다. 임베딩은 Hugging Face 무료 로컬 모델(sentence-transformers)로, AI 답변은 Google Gemini 무료 tier로 전환하여 전체 시스템을 비용 0원으로 설계하였습니다.

핵심 설계 결정
· 임베딩: 유료 API 대신 로컬 sentence-transformers → 비용 0원, 인터넷 불필요
· 검색: 키워드 단독이 아닌 BM25+벡터 하이브리드 방식 → 정확도 향상
· AI 답변: Gemini 무료 tier 활용 → 운영 비용 최소화
· DB: PostgreSQL 하나로 전문검색+벡터 검색 통합 → 인프라 단순화

Phase 2 — 데이터 수집 (크롤링)

국립중앙도서관 사서문답 서비스에서 591페이지 × 10건 = 5,910건을 수집 대상으로 설정하고 자동 크롤러를 개발하였습니다. 초기에는 HTML 엔티티 파싱 오류로 3건이 누락되었으나, lxml 파서 전환으로 최종 5,910건 전체를 완전 수집하였습니다.

항목내용
수집 대상국립중앙도서관 사서문답 서비스 (nl.go.kr)
수집 필드질문, 답변, 도서관명, 카테고리, 조회수, 출처 URL, 키(ackRecKey)
최종 수집량5,910건 (100%) 업데이트

Phase 3 — DB 구축 및 임베딩

PostgreSQL 17에 pgvector 확장을 설치하여 벡터 저장 및 코사인 유사도 검색 환경을 구축하였습니다. paraphrase-multilingual-MiniLM-L12-v2 모델로 각 Q&A의 질문+답변 텍스트를 384차원 벡터로 변환하여 저장하였습니다. 5,910건 전체 임베딩 완료 후 IVFFlat 인덱스를 생성하여 검색 속도를 최적화하였습니다.

Phase 4 — 하이브리드 검색 구현

쿼리 유형BM25 비중벡터 비중판별 조건
기본 (기본값)35%65%기타 모든 질문
주제 탐색형20%80%'주제', '관련', '추천' 등 포함
고유명사형70%30%ISBN, 저자명, 출판사 등 포함
가중치 설계 근거

① 기본값: BM25 35% + 벡터 65%
국립중앙도서관 사서문답 Q&A의 대부분은 "대출 연장은 어떻게 하나요", "청구기호 읽는 방법"처럼 특정 키워드보다 질문의 의도·맥락이 중요한 개념·절차형 질문입니다. 벡터 검색은 표현이 달라도 의미가 같은 문장을 잘 찾는 반면, BM25는 동일 단어가 없으면 관련 문서를 놓칩니다. 따라서 의미 유사도(벡터)에 더 높은 비중을 두되, 특정 용어가 포함된 질문에도 대응할 수 있도록 BM25를 35% 보완합니다.
→ 왜 70%나 60%가 아닌 65%인가: BM25를 35% 유지해야 '청구기호', '소장', '사서' 등 도서관 전문 용어처럼 표현 변이가 적은 단어에서 정밀도를 확보할 수 있습니다. 완전한 벡터 중심으로 가면 범용 다국어 임베딩 모델의 특성상 전문 용어에서 유사도 오차가 발생할 수 있어, 65%를 두 방식의 균형점으로 설정하였습니다. 향후 실사용 쿼리 로그를 수집하면 그리드 서치로 최적값을 정밀화할 수 있습니다.

② 고유명사형: BM25 70% + 벡터 30%
ISBN, 저자명, 출판사처럼 고유명사·식별자가 포함된 질문은 정확한 토큰 일치가 핵심입니다. 벡터 임베딩 모델은 고유명사를 의미 공간에서 일반화하는 경향이 있어, 특정 저자명이 비슷한 장르의 다른 저자와 가깝게 매핑될 수 있습니다. 반면 BM25는 입력된 문자열을 그대로 토큰으로 비교하므로 고유명사 검색에 유리합니다.
→ 왜 80%나 100%가 아닌 70%인가: 벡터 30%를 유지함으로써 해당 저자의 분야·장르와 관련된 문서를 의미적으로 보완 검색할 수 있습니다. BM25를 100%로 올리면 질문의 맥락과 의도를 완전히 무시하게 되어, 예를 들어 저자명 오기재 시 관련 결과를 전혀 찾지 못하는 취약점이 생깁니다. 70%는 정확도와 맥락 보완 사이의 실용적 균형점입니다.

③ 주제 탐색형: BM25 20% + 벡터 80%
"한국 현대문학 관련 자료 추천해줘"처럼 넓은 주제 영역을 탐색하는 질문은 특정 키워드가 없어도 의미적으로 유사한 Q&A를 최대한 많이 찾아야 합니다. 이 경우 벡터 검색의 시맨틱 커버리지가 가장 중요하므로 벡터 비중을 80%까지 높입니다.
→ 왜 90%가 아닌 80%인가: 주제 탐색형 질문에도 '한국', '현대문학'처럼 BM25가 정확히 잡아야 할 핵심 키워드가 포함됩니다. BM25를 20% 유지하면 이런 키워드가 일치하는 문서에 가점이 가지만, 90%로 올리면 BM25 기여가 사실상 노이즈 수준으로 줄어 키워드 정밀도가 손실됩니다. 현 설계값은 합리적 시작점이며, 실사용 데이터 기반 최적화 여지를 남겨 두었습니다.

오류 1 — 크롤러 파라미터 오류

[문제] 국립중앙도서관 목록 API 호출 시 페이지 파라미터를 pageNum으로 설정. 실제 API는 page 파라미터를 사용하여 매 요청이 1페이지만 반복 수집됨. 5,910건 수집 시도 중 동일한 10건이 591회 반복 저장되는 오류 발생.

[해결] 브라우저 개발자 도구로 실제 요청 파라미터를 확인하여 page로 수정. 수집 데이터 전체 삭제 후 재수집 진행.

오류 2 — views 필드 파싱 오류

[문제] 조회수(views) 필드 값이 '1,234' 형태의 천단위 쉼표 포함 문자열로 수집됨. int('1,234') 변환 시 ValueError 발생, 해당 건 전체 DB 적재 실패.

[해결] int 변환 직전에 .replace(',', '') 전처리를 추가하여 쉼표 제거 후 변환.

오류 3 — pgvector 설치 복잡성 (Windows 환경)

[문제] pgvector는 PostgreSQL C 확장으로, Windows에서는 Visual Studio Build Tools 및 PostgreSQL 개발 헤더 경로 설정 필요. 공식 문서대로 진행 시 nmake 빌드 오류 반복 발생.

[해결] GitHub Releases에서 Windows용 사전 빌드 바이너리를 다운로드. C:\Temp\pgvector_extract에 압축 해제 후 .dll/.control 파일을 PostgreSQL 설치 디렉터리에 수동 복사하여 CREATE EXTENSION vector 성공.

오류 4 — Gemini 전환 코드 혼재

[문제] Claude → Gemini 전환 도중 작업이 중단되어 google.generativeai import와 Anthropic SDK 스타일 호출이 동일 파일 내 혼재. 서버 실행 시 AttributeError 발생, /api/ask 엔드포인트 응답 불가.

[해결] generate() 함수 전면 재작성. google.genai 최신 SDK의 client.models.generate_content_stream() 방식으로 교체.

오류 5 — 패키지 미설치

[문제] google.generativeai 임포트 시 ModuleNotFoundError 발생. 가상환경 초기 설정 시 requirements.txt에 해당 패키지가 누락된 상태.

[해결] pip install google-generativeai 로 즉시 설치. 이후 requirements.txt에 추가하여 재현 방지.

오류 6 — SDK deprecated 경고

[문제] google.generativeai 라이브러리가 공식 deprecated 처리됨. 일부 신규 모델 파라미터 미지원으로 향후 버전에서 완전 제거될 위험.

[해결] google.genai (google-genai 패키지) 최신 SDK로 전면 전환. Client 초기화 방식 및 스트리밍 호출 구조 전체 재작성 완료.

오류 7 — Gemini API 쿼터 초과 (429)

[문제] gemini-2.0-flash 무료 tier 일일 요청 한도 초과로 RESOURCE_EXHAUSTED 오류 발생. 새 API 키를 발급해도 동일 GCP 프로젝트 내에서는 쿼터가 공유되어 동일 오류 반복.

[해결] gemini-2.5-flash 모델(별도 독립 쿼터)로 변경 + 신규 GCP 프로젝트에서 API 키 재발급. 쿼터 초과 시 검색 결과를 직접 반환하는 폴백 로직도 추가 구현.

오류 8 — 구버전 서버 프로세스 잔류

[문제] uvicorn --reload 옵션으로 재시작 후에도 구버전 reloader 프로세스가 포트 8000 점유. 코드 수정이 전혀 반영되지 않아 /api/ask가 항상 동일 메시지만 반환.

[해결] Get-Process python* | Stop-Process -Force로 모든 Python 프로세스 강제 종료 후 --reload 없이 재시작. 운영 시 --reload 옵션 미사용 권장.

오류 9 — 한글 인코딩 출력 오류

[문제] Windows 콘솔 기본 인코딩이 CP949인 환경에서 Python print()로 UTF-8 한글 출력 시 UnicodeEncodeError 발생.

[해결] sys.stdout.reconfigure(encoding='utf-8', errors='replace') 방식으로 해결.

오류 10 — HTML 엔티티 파싱 오류

[문제] 일부 레코드(3건)의 HTML에 세미콜론이 누락된 불량 HTML 엔티티(&#039doopedia...)가 포함. Python 3.14의 html.parser가 int() 변환 시 ValueError 발생. BeautifulSoup 객체 생성 단계에서 예외가 발생하여 기존 try-except로도 포착 불가.

[해결] BeautifulSoup(html, "lxml")으로 파서 전환. lxml은 불량 HTML에 관용적으로 처리하여 정상 파싱 가능. 누락 3건 완전 수집 → 총 5,910건 (100%) 달성.

오류 11 — 발표 자동실행 배치파일 인코딩 오류 신규

[문제] 발표 당일 서버·ngrok·Expo를 일괄 시작하는 발표시작.bat를 더블클릭하였으나 창이 나타났다 즉시 사라지고 아무것도 실행되지 않음. 원인 분석 결과, 배치 파일이 UTF-8 인코딩으로 저장되어 있었으나 Windows cmd.exe가 시스템 기본 코드페이지(CP949)로 파일을 읽으면서 한글 UTF-8 바이트 시퀀스가 명령어로 잘못 파싱됨. 예: AI사서 FastAPI의 한글 바이트 경계에서 문자열이 분리되어 tAPI가 독립 명령어로 인식되어 오류 발생. 같은 이유로 발표시작.ps1 전환 후에도 PowerShell 5.1이 BOM 없는 UTF-8 파일을 CP949로 읽어 한글 경로(C:\VIBE\9. AI사서응답시스템)가 깨져 Start-Process -WorkingDirectory 디렉터리 탐색 실패.

[해결 1단계] PowerShell의 [System.IO.File]::WriteAllText()에 [System.Text.UTF8Encoding]::new($true)를 지정하여 UTF-8 BOM(EF BB BF) 포함으로 PS1 파일을 재저장. PowerShell 5.1은 BOM을 감지하면 UTF-8로 파일을 올바르게 읽으므로 한글 경로 및 한글 문자열 처리 문제 해결.

[해결 2단계] bat 파일 내용에 "C:\VIBE\발표시작.ps1"이라는 한글 경로가 남아 있어, cmd.exe가 다시 해당 바이트를 CP949로 오독하여 창이 즉시 종료되는 문제 재발. PS1 파일을 순수 영문 이름 start.ps1으로 복사하고, bat 파일을 powershell -ExecutionPolicy Bypass -File "C:\VIBE\start.ps1" 한 줄(완전 ASCII)로 변경.

[해결 3단계] 파일 정리 목적으로 발표시작.bat과 start.ps1을 C:\VIBE\9. AI사서응답시스템\으로 이동한 후, bat 파일의 경로를 "C:\VIBE\9. AI사서응답시스템\start.ps1"으로 하드코딩하였으나 폴더명의 한글(사서응답시스템)이 다시 bat 파일 내에 포함되어 동일 오류 3회 재발. cmd 내장 변수 %~dp0(실행 중인 bat 파일의 폴더 경로를 런타임에 자동 해석)을 사용하여 powershell -ExecutionPolicy Bypass -File "%~dp0start.ps1"으로 변경. bat 파일 텍스트에 한글이 전혀 없으면서도 올바른 경로를 참조하여 최종 정상 실행 확인.

6. 개선 방안

6.1 데이터 품질 — 한국어 특화 임베딩 모델 도입
• BGE-M3, KoSimCSE 등 한국어 특화 모델로 교체 시 도서관 전문 용어 검색 정밀도 개선 가능
필요성  현재 paraphrase-multilingual-MiniLM은 범용 다국어 모델로, 도서관 전문 용어·한국어 어휘 변이에서 의미 유사도 정확도가 제한됨. '사서', '소장', '청구기호' 등 고유 개념에서 벡터 거리 오차 발생 가능.
단점  전체 5,910건 재임베딩 수 시간 소요, 모델 파일 수 GB 저장 공간 부담, 한국어 특화 모델일수록 영어·다국어 질문 대응력 저하, 라이선스 확인 필요.
6.2 검색 품질 — 도서관 용어 동의어 사전 구축
• '대출=빌리기', '반납=돌려주기' 등 일상어-전문용어 매핑으로 BM25 키워드 검색 재현율 향상
필요성  이용자는 전문 용어에 익숙하지 않아 일상어로 질문하면 키워드 검색에서 결과가 누락됨. 벡터 검색이 보완하고 있으나 고유명사·약어 처리는 한계가 있음.
단점  사전 초기 구축 및 유지보수에 도메인 전문가 투입 필요, 과잉 확장 시 무관한 결과 혼입 가능, 신조어·비표준 표현의 주기적 갱신 부담.
6.3 AI 답변 품질 — 응답 캐싱 도입
• Redis 또는 DB에 빈출 질문 상위 50건 답변 사전 저장 → Gemini API 호출 횟수 절감
필요성  동일·유사 질문이 반복될 때마다 Gemini API를 호출하면 무료 쿼터 소진이 가속됨. 캐싱 도입 시 응답 속도 향상과 쿼터 절약을 동시에 달성 가능.
단점  Redis 서버 별도 구축·운영 필요(추가 인프라), 캐시 유효기간 및 무효화 로직 관리 복잡, AI 답변 품질 개선 시 기존 캐시 전체 재생성 필요.
6.4 시스템 안정성 — 프로세스 자동 관리
